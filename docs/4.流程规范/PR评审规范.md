# PR 评审规范

> 规范代码评审流程，确保代码质量和知识共享。

---

## 📝 PR 创建规范

### 1. PR 标题

```
{类型}: {简短描述}

示例:
feat: 添加用户中心页面
fix: 修复购物车数量显示异常
```

### 2. PR 描述模板

```markdown
## 📋 变更内容

- 描述本次变更的内容

## 🧪 测试情况

- [ ] 本地测试通过
- [ ] 已在开发环境验证

## 📸 截图/录屏

（如有 UI 变更，请附上截图）

## 🔗 关联 Issue

#123

## ⚠️ 注意事项

（如有特殊说明，在此填写）
```

---

## 👀 Reviewer 职责

### 1. Review 检查清单

- [ ] 代码逻辑是否正确
- [ ] 是否符合代码风格规范
- [ ] 是否有潜在的性能问题
- [ ] 是否有安全风险
- [ ] 是否添加了必要的注释
- [ ] 单元测试是否覆盖
- [ ] 变更是否影响其他功能

### 2. Review 态度

- ✅ **鼓励型**: "这个思路很好，建议可以..."
- ✅ **建设型**: "这里有个更好的方案..."
- ❌ **避免**: "为什么要这样写？"

### 3. 评论格式

```markdown
> **问题描述**  
> 建议的解决方案（可选）
>
> {Reviewer} - {日期}

---

位置: `pages/index/index.ts:45`

建议: 这里的异常处理可以更完善...

[MUST] - 必须修改
[SHOULD] - 强烈建议修改
[NIT] - 小问题，不强制
[QUESTION] - 疑问，需要澄清
[PRAISE] - 代码写得好！
```

---

## 🔄 评审流程

```
创建 PR → 自动检查 → Reviewer Review 
                          ↓
                    通过? → 是 → 合并
                          ↓ 否
                      修复 → 重新 Review
```

### 1. 自动化检查 (CI)

- TypeScript 编译检查
- ESLint 代码规范检查
- 单元测试

### 2. 人工 Review

- 至少 1 人 Review (develop)
- 至少 2 人 Review (main)

### 3. 合并

- Squash Merge 或 Merge (团队协商决定)
- 删除源分支

---

## ⚠️ 评审优先级

| 优先级 | 说明 | 时限 |
|-------|------|-----|
| P0 | 紧急修复 | 2 小时内 Review |
| P1 | 新功能 | 24 小时内 Review |
| P2 | 优化/重构 | 48 小时内 Review |
| P3 | 文档更新 | 72 小时内 Review |
