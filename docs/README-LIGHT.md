# 轻量级 SDD 规范 (小团队版)

> 专为 3-5 人小团队设计，保留核心、减少负担

---

## 🎯 核心原则

```
✅ 最小化文档 → 只写必要的
✅ 快速迭代   → 边做边调整  
✅ 保持一致   → 核心规范必须遵守
❌ 不要过度设计 → 不要为了规范而规范
```

---

## 📋 简化流程

### 原来的 SDD 流程（6步）
```
需求 → 完整规格文档 → 评审 → 开发 → 自测 → Code Review
```

### 轻量级 SDD 流程（3步）
```
需求 → 简洁规格 → 开发 → 自检 ✅ 完成！
        (可选)   (简单备注即可)
```

---

## 📝 规格文档简化

### 原来：完整 SPEC.md（10+ 部分）
### 现在：功能卡片（3 部分）

```markdown
# [功能名称]

## 🎯 做什么
一句话描述

## 📋 要点
- [ ] 要点1
- [ ] 要点2

## 🔗 关联
- 关联页面/组件
- 需修改的接口
```

**位置**: 直接放在页面目录下 `pages/xxx/NOTES.md`

---

## 📐 核心规范保留

### 1. 目录结构不变 ✅
```
pages/          # 页面
components/     # 全局组件
services/       # 服务层
```

### 2. 命名规范不变 ✅
```
页面目录: kebab-case    → goods-detail/
组件: kebab-case        → goods-card/
变量: camelCase         → userInfo
类型: PascalCase        → IUserData
```

### 3. TypeScript 类型必须 ✅
```typescript
// 至少要有接口定义
interface IGoodsItem {
  id: string;
  name: string;
  price: number;
}
```

## 🔧 协作规则（5人团队）

| 场景 | 规则 |
|-----|------|
| 新功能开发 | 在对应页面创建 NOTES.md |
| 组件复用 | 先问大家，避免重复 |
| 接口约定 | 口头或群里确认，更新 API 文档 |
| 代码评审 | 简单的 check，10分钟快速过 |
| 合并主分支 | 直接合并，不需要复杂 PR |

---

## ⏰ 会议节奏

| 类型 | 时长 | 频率 | 内容 |
|-----|------|------|-----|
| 站会 | 5-10min | 每天 | 进度、阻塞、计划 |
| 同步会 | 15min | 每周 | 规格确认、方案讨论 |

## 📄 文档清单（小团队精简版）

```
docs/
├── 0.项目规范/
│   ├── 代码风格指南.md       # ✅ 必须 - 保持代码一致
│   └── Git规范.md           # ✅ 必须 - 分支和提交约定
│
├── 1.产品规格/
│   ├── 产品总览.md          # ✅ 必须 - 功能列表
│   └── pages/               # 页面级 NOTES.md（按需）
│
├── 2.技术架构/
│   ├── 目录结构.md          # ✅ 必须 - 避免混乱
│   └── API文档.md           # ✅ 必须 - 接口约定
│
└── 3.组件规范/
    └── 组件清单.md          # ✅ 必须 - 复用组件
```

**去掉**: 详细的 PR 评审规范、复杂的发布流程、完整的组件开发规范
